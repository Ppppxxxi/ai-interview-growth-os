# 推送到上线校验流程

这份流程用于每次功能迭代后，从本地检查、推送 GitHub、触发部署，到线上验证是否可以保留。本项目当前以 Vercel 部署为主，真实个人求职材料只允许保存在浏览器本地数据中，不进入仓库。

## 0. 产品改动判断

每次准备上线前，先确认本次改动是否符合当前产品路径。

当前 v0.3.1 之后的主路径是：

```text
聚焦模式：粘贴 JD、面试记录或模拟面试对话
  -> 解析材料
  -> 确认识别结果
  -> 生成下次可用回答
  -> 保存到我的回答库

工作台：查看历史岗位、回答库、面试复盘、下次训练题
```

应当符合：

- 主入口使用用户语言：`粘贴 JD、面试记录或模拟面试对话`。
- 标准格式入口只作为副入口：`按标准格式分别填写` 或 `高级导入`。
- 每个阶段只保留一个主按钮。
- 生成前不要出现“保存为资产”这类内部动作。
- 主流程文案优先使用“下次可用回答 / 我的回答库”。
- “回答资产”可以保留在详情、标签、文档和内部模型中，不放在首次使用主流程里教育用户。
- 生成前只做草稿级识别反馈，不自动生成最终复盘。
- 工作台用于沉淀管理，不替代聚焦流程。

不应当出现：

- 首屏同时堆叠岗位档案、回答库、复盘总结、训练题、多个输入入口。
- `0 场记录`、`待确认`、`待生成` 这类负反馈占据主视觉。
- 两个同等级主按钮让用户分不清下一步。
- 真实公司、真实面试经历、真实 JD、个人联系方式进入代码、README、截图或测试数据。

## 1. 本地改动检查

在项目 worktree 中执行：

```powershell
git status --short --branch
git diff --stat
```

检查点：

- 确认当前分支正确。
- 确认没有构建产物 `dist/` 被误提交。
- 确认没有无关文件混入。
- 确认本次改动范围和版本目标一致。

如果涉及公开页面、示例数据、测试数据或文档，必须做脱敏搜索：

```powershell
rg -n "真实公司名|真实岗位名|手机号|邮箱|微信|身份证|真实面试关键词" src docs README.md
```

如果搜索到真实信息，先替换为虚构公司和泛化业务场景，再继续。

## 2. 本地质量验证

每次准备推送前执行：

```powershell
npm.cmd test
npm.cmd run build
```

通过标准：

- `npm.cmd test` 全部通过。
- `npm.cmd run build` 成功生成生产包。
- 如果只是纯文档改动，可以不跑完整构建，但发布前合并到主分支仍建议跑一次。

如果失败：

- 先修复失败项。
- 不要带着失败测试推送部署。
- 如果失败来自历史问题而非本次改动，需要在发布说明里记录风险。

## 3. 文档同步检查

功能或体验版本需要同步：

- `README.md`
- `docs/ROADMAP.md`，如果路线发生变化
- 对应版本的 `docs/RELEASE_NOTES_V*.md`
- `docs/agents/agent-contracts.md`，如果涉及 Agent 输入输出或 LLM 替换边界
- `docs/PUBLIC_REPO_RISK.md`，如果涉及公开风险

v0.3.1 之后，体验类发布说明至少写清：

- 本版本解决什么真实使用问题。
- 改了哪些页面和流程。
- 哪些能力仍然不做。
- 验证命令。

## 4. 提交前检查

执行：

```powershell
git diff --check
git status --short
```

然后暂存并提交：

```powershell
git add <changed-files>
git commit -m "<type>: <short summary>"
```

提交信息建议：

- `feat:` 新增用户可感知能力。
- `fix:` 修复错误或明显体验问题。
- `docs:` 仅文档变化。
- `chore:` 数据脱敏、版本号、维护类改动。

示例：

```powershell
git commit -m "feat: calibrate v0.3.1 onboarding workflow"
git commit -m "chore: anonymize material import sample data"
git commit -m "docs: add deployment validation workflow"
```

## 5. 推送 GitHub

确认分支领先远程：

```powershell
git status --short --branch
git log -3 --oneline --decorate
```

推送当前分支：

```powershell
git push origin <branch-name>
```

如果这是功能分支：

- 在 GitHub 上创建或更新 PR。
- 确认 PR 只包含本次目标相关提交。
- 合并前检查 GitHub Actions 或自动检查是否通过。

如果直接推送主分支：

- 确认这是已验证的小改动。
- 推送后立即进入 Vercel 检查部署状态。

## 6. Vercel 部署检查

进入 Vercel 项目页面，确认：

- 最新 commit 被识别。
- Build 成功。
- Production deployment 指向最新 commit。
- 线上 URL 可以打开。

如果部署失败：

- 先看 Vercel build log。
- 若是构建失败，回到本地复现 `npm.cmd run build`。
- 若是环境问题，确认是否误引入了真实 LLM、环境变量或网络依赖。
- 不要在生产站保留失败版本。

## 7. 线上功能校验

打开线上地址后，用无真实隐私的材料跑一遍。

### A. 首次进入

检查：

- 首页是否清楚表达“粘贴材料 -> 检查草稿 -> 保存回答”。
- CTA 是否只有两个：开始真实使用 / 使用示例体验。
- 示例数据是否有明显标识。
- 没有真实个人信息或真实面试公司。

### B. 生成本场回答

清空本地数据后检查：

- 进入后首屏聚焦“粘贴材料”。
- 未生成前右侧只显示下一步提示。
- 不出现大面积 `0 场记录`、`待确认`、`待生成`。
- `高级导入` 是折叠入口，不抢主流程。
- 点击“填入材料示例”后可以生成可确认草稿。
- 点击主按钮后，草稿能展示问题、原回答、具体问题、修改建议和下次可用回答。
- 保存后能生成面试记录和回答。

### C. 我的回答库

检查：

- 保存后的回答能在“我的回答库”中找到。
- 搜索和筛选可以用。
- 回答来源岗位、来源面试、适用问题显示清楚。
- 记录使用反馈不会报错。

### D. 面试复盘

检查：

- 有数据时展示短板和沉淀情况。
- 数据少时不强行展示复杂图谱。
- 训练建议不喧宾夺主。

### E. 本地数据

检查：

- 刷新页面后本地数据仍在。
- 清空本地数据后回到首页。
- 恢复示例不会混入真实数据。

## 8. 上线验收标准

满足以下条件才算本次上线可保留：

- 用户 30 秒内能理解：导入材料后，系统会生成下次可用回答。
- 主流程没有两个并列入口抢注意力。
- 每个阶段只有一个明显主按钮。
- 未生成前没有大面积负反馈状态。
- 示例数据已脱敏。
- `npm.cmd test` 和 `npm.cmd run build` 通过。
- 线上核心流程能跑通：粘贴材料 -> 生成草稿 -> 保存回答 -> 回答库可检索。

## 9. 回滚判断

出现以下情况应回滚或立即修复：

- 线上页面打不开。
- 主流程按钮不可用。
- 保存后数据丢失或无法进入回答库。
- 公开页面出现真实公司、真实面试内容或个人隐私。
- 新版本让用户更难开始，例如首屏重新变成多表单配置台。

回滚方式：

- 优先在 Vercel 中回滚到上一版稳定部署。
- 同时在 GitHub 开新修复提交。
- 不建议用破坏性 Git 操作处理已推送历史。
