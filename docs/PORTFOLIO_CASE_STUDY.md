# Portfolio Case Study：AI 面试成长 OS

## 1. 项目一句话

AI 面试成长 OS 是面向产品经理求职者的本地面试复盘与回答资产沉淀工具。它不替代 ChatGPT、Claude 或 Codex 做模拟面试，而是作为第二层工作台，把分散在外部 AI 对话里的面试经验结构化沉淀，并在同岗位多轮或同方向类似岗位中复用。

## 2. 用户痛点

目标用户是高频求职中的产品经理候选人，尤其是同时准备多个岗位、频繁使用 AI 做模拟面试的人。

典型问题：

- 每个岗位需要单独开一个 AI 对话 session，跨岗位信息割裂。
- JD 分析、问题参考、模拟回答、AI 点评和复盘材料混在长对话里，后续很难检索。
- 一场面试失败后，该岗位可能不会继续，但其中暴露的短板、被追问的角度和可复用表达仍然有价值。
- 现有 AI 模拟面试更像一次性生成工具，缺少“复盘 -> 改写 -> 资产 -> 下次复用”的长期闭环。

## 3. 产品定位

最初方向是“AI 面试成长 OS / 长期面试成长系统”，但第一阶段不能做成大而全的模拟面试、求职 CRM 或能力测评工具。因此 MVP 收敛为：

```text
粘贴 JD
  -> 粘贴外部 AI 模拟面试对话
  -> 解析原问题、原回答、追问和点评
  -> 生成结构化复盘
  -> 生成可编辑优化回答
  -> 保存为回答资产
  -> 在下次同岗位多轮或同方向类似问题中复用
```

核心差异化不是“再做一个模拟面试助手”，而是沉淀外部 AI 对话产生的经验。

## 4. MVP 取舍

### 保留

- 岗位档案：每个岗位作为准备上下文。
- 外部对话导入：支持用户继续使用熟悉的 AI 工具做模拟面试。
- 结构化复盘：把长对话压缩为问题、回答、短板和下一步准备。
- 回答资产库：作为核心产出，而不是把评分当成主要卖点。
- 使用反馈和版本管理：让资产能随着真实面试继续迭代。

### 降级

- 内置模拟面试：作为辅助入口，不作为主流程。
- 能力图谱：只展示重复短板和资产沉淀，不做复杂均分图。
- 训练计划：先简化为考前优先练习问题。
- 经历匹配：保留为岗位准备辅助，不作为冷启动门槛。

### 暂不做

- 登录、云同步、多人协作。
- 真实 LLM API。
- 自动投递、岗位搜索、求职 CRM。
- 大规模统计分析。

## 5. 信息架构

当前版本保留三个主视图：

1. **岗位工作台**
   - 聚合 JD、面试记录、导入对话、复盘报告、该岗位关联回答资产。
   - 第一触点是“粘贴 JD + 粘贴面试对话 -> 生成复盘与优化回答”。

2. **回答资产库**
   - 跨岗位方向检索回答资产。
   - 支持搜索、筛选、使用反馈、回答版本管理和 Markdown 导出。

3. **全局成长**
   - 作为长期视图，展示重复短板、资产沉淀和复用范围。
   - 不在数据不足时强行展示复杂能力图谱。

## 6. 当前 v0.2 功能闭环

### 岗位工作台

- 新建、编辑、删除岗位档案。
- 删除岗位时同步清理关联面试记录、复盘和回答资产。
- 预览并编辑外部对话解析结果。
- 保存多场面试记录。
- 切换历史面试记录并查看详情。
- 导出单场面试复盘和岗位考前准备包。

### 回答资产库

- 关键词搜索。
- 按岗位方向、问题类型、能力短板、来源岗位、置信度和使用状态筛选。
- 记录使用反馈：是否有效、面试官追问、使用后复盘。
- 保存回答新版本，保留历史版本，并支持恢复历史版本。
- 导出回答资产合集。

### 本地持久化

- 使用 `localStorage` 保存个人数据。
- GitHub 仓库只包含 mock 示例数据。
- 真实 JD 和真实面试材料不会进入仓库。

## 7. AI Agent 设计

当前版本使用 deterministic mock agent，保证本地稳定运行、无需 API Key、无需上传真实数据。

真实 LLM 版本会优先替换以下模块：

1. `importParser`：解析外部 AI 面试对话。
2. `answerAssetGenerator`：生成更贴合用户真实经历的优化回答。
3. `reviewEvaluator`：生成结构化复盘。
4. `jdAnalyst`：分析 JD 画像。
5. `growthPlanner`：识别跨场次重复短板。

每个 Agent 都应保持稳定 JSON 输入输出，并设置失败兜底策略。用户必须能编辑和确认 AI 结果后再保存为回答资产。

## 8. AI 产品设计判断

这个项目刻意避免把 AI 输出当成最终答案，而是把 AI 结果放进一个可控工作流：

- AI 负责生成草稿和结构化建议。
- 用户负责确认回答是否符合真实经历。
- 系统负责保留来源、版本、反馈和复用场景。
- 低置信或不符合经历的回答不应直接进入资产库。

对面试准备这种高风险场景，用户控制权比“自动生成更多内容”更重要。

## 9. 可验证指标

如果进入真实产品验证，可以优先看：

- 单场面试复盘完成率。
- 优化回答保存率。
- 回答资产搜索使用次数。
- 回答资产被标记为“已使用”的比例。
- 使用后仍被标记“需要重写”的比例。
- 同一能力短板在后续面试中的重复出现次数。

这些指标比单纯的 1-5 分评分更能说明产品是否真的帮助用户准备下一场面试。

## 10. 后续路线

### v0.3 LLM 接入实验

- 优先接入 `importParser` 和 `answerAssetGenerator`。
- 增加 JSON schema 校验。
- 失败时回退 deterministic mock agent。
- 增加 prompt 版本管理和低置信提示。

### v0.4 可信工作流

- 增加本地脱敏提示。
- 增加“采纳 / 修改 / 放弃”记录。
- 增加复盘质量反馈，反向优化 prompt。

### v1.0 真实可用工具

- 登录与云同步。
- 多设备访问。
- 简历经历库和回答资产联动。
- 可配置模型供应商和隐私策略。

## 11. 作品集表达

这个项目适合用于 AI 产品经理岗位作品集，重点展示：

- 从个人真实痛点抽象出高频用户场景。
- 把“模拟面试”收敛为“面试经验结构化沉淀与复用”。
- 在 AI 生成能力之外设计用户确认、版本、反馈和来源追溯机制。
- 明确 mock agent 与真实 LLM 的边界。
- 用可运行 Demo 证明 MVP 闭环，而不只是停留在 PRD。
